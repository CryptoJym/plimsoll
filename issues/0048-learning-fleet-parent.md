# 0048 — Parent: evidence-backed learning and fleet-ready Plimsoll

GitHub mirror: https://github.com/CryptoJym/plimsoll/issues/96

## TL;DR
- Turn Plimsoll from a strong telemetry/accounting collector into a coverage-aware learning system: conserved project allocation, immutable delivery lineage, correction/error metrics, explicit technique exposures, and reviewable skill-candidate evidence.
- Make the same accepted collector safe and supportable for company Macs: pinned packaged runtime, transactional enrollment, truthful doctor, update/rollback/uninstall, device identity/revocation, and a two-machine E2E.
- Keep the 80/20 boundary: deterministic facts and descriptive own-data analysis locally; comparative/prescriptive analysis in the hosted product; durable skills remain owner-reviewed doctrine artifacts.
- No live collector activation, ledger migration, cloud deploy, credential movement, or autonomous skill publication is authorized by this issue.

## Scope
Parent program for public collector issues 0049–0057 and paired hosted issues under `CryptoJym/plimsoll-cloud#24`. Each child is independently shippable and proof-gated.

## Context
- Accepted collector architecture: `main@883c71bc80e3519f700c07af04ec12d9696a89fb`.
- Real ledger read-only audit: 4,810,030 events; 7,323 sessions; 99,435,009,371 direct input+output tokens; 87,503,371,122 project-attributed; 225 sessions span multiple repos and carry 19,771,520,024 direct tokens.
- Current reports provide model/action/session/repo/account summaries, tokens/cost per merged PR, and VDY v1/v2.
- Current learning schemas (`Intervention`, `SkillRecommendation`, `BehaviorDelta`) are dormant contracts with no evidence or promotion engine.
- Current public package is `@plimsoll/cli@0.6.0`, older than accepted main; the source installer is broken because it omits the required `--dev` flag.

## Acceptance Criteria
- [ ] Project/work-unit allocations conserve all token classes and known cost; no event is charged more than once.
- [ ] First-pass, correction-loop, time-to-green, stable-delivery, and rework metrics use immutable timelines and expose completeness/maturity.
- [ ] Technique results are labeled observed/suggestive/associated/causal; causal requires an explicit experiment assignment.
- [ ] Skill candidates contain provenance, counterexamples, uncertainty, expiry, dedupe result, rollback, independent verification, and owner approval; no global skill tree is written automatically.
- [ ] Fresh pinned install → config preview/apply → ready collector → join → first token event succeeds on supported Mac architectures.
- [ ] Two machines use unique credentials/device identities; reassign/revoke/rollback/uninstall paths pass without moving credentials.
- [ ] Learning and fleet E2E remain inside fixed CPU, memory, I/O, request-body, cardinality, and background-cadence budgets.
- [ ] Final exact-head proof covers privacy, allocation conservation, source authenticity, installer rollback, and live-state untouched unless a separate rollout is approved.

## Operational Boundaries
- Metadata-only for company rollout; evidence mode is out of scope until an encrypted vault and policy enforcement exist.
- Raw prompts, responses, commands, arguments, paths, diffs, secrets, and PII never enter metric or skill evidence packets.
- Credentials never move between machines or workspaces.
- LLMs may summarize aggregate evidence and nominate hypotheses; deterministic code computes canonical metrics; humans approve doctrine/skills.
- No individual ranking below policy cohort gates; no composite developer score.
